# V10 C-Stage Overview (C0.1 → C1.2) / C阶段总览（人话版）— 2025-12-22

## 这条线一句话在做什么？

**我们做的不是赚钱，也不是“策略研究本身”。**  
我们是在给系统装一套**可长期运行的事故记录仪**：确保进入真实执行环境后，任何异常都能**落盘、复现、回放、审计**，而不是靠运气和故事解释。

---

## 为什么必须做？（否则会怎样）

进入执行接口世界（demo/live）后会遇到三类“工程黑洞”：

- **I（持仓）不可信**：交易所/库返回的 `positions` 可能空、慢、错。没有 fallback 就会“见鬼”。
- **M（摩擦）不可信**：延迟、部分成交、错误、费用、成交干旱……这些决定生死，但如果不落盘，就只能凭感觉争论。
- **长跑不可持续**：日志刷屏、文件无限增长、STOP 不可控 → 无法“睡觉级别”长跑，更别谈 VPS 试运行。

---

## C0.x：先证明“证据链能活着”

### C0.5 / G4.5（执行接口世界能落盘）
- 证明 `okx_demo_api` 下 **api_calls > 0**（真实交互）
- 对齐证据包可落盘：alignment report + raw samples（不靠终端口述）

### C0.7（I：持仓可得）
- demo 下 `positions` 不可靠时，不假装可得
- 用“成交证据”重建持仓 → 产出 `positions_snapshot.json`
- manifest 口径升级为 `reconstructed_from_fills`

### C0.8（I：持仓可追溯）
- 不止有“重建结果”，还要有“重建依据”
- 产出 `positions_reconstruction_raw.json`（脱敏）+ hash

### C0.9（M：摩擦可追溯）
- 产出 `m_execution_raw.json`（脱敏）+ hash
- 以后讨论“想平平不掉”的痛苦，有证据可引用

## C1.x：再证明“能长期跑（睡觉级别）”

### C1.0（1小时）
- 证据链 8件套在 1 小时内不断
- STOP 语义可审计（`interrupted` + `stop_*` 落盘）

### C1.1（raw 增长控制）
- raw 证据会增长，但必须可控：**分片 + 索引 + hash + manifest 引用**
- 防止长跑变成“磁盘黑天鹅”

### C1.2（6小时）
- 验证“睡一觉级别”多小时运行仍稳定：
  - 证据链不断
  - 分片索引不崩
  - raw 增长可控

---

## 最终我们得到了什么能力？

我们现在能回答三件事：

1) **系统到底做了什么？**（证据链落盘）
2) **系统为什么会这样？**（I/M raw 可追溯）
3) **系统能不能长期活着？**（长跑 + 增长控制 + STOP）

这就是 C阶段在做的事。
